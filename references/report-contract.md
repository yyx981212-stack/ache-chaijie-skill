# 可视化报告输出契约

进入任何正式账号分析时必须读取，不等待用户二次要求 HTML/可视化报告。内容契约高于宿主 App 的默认排版，UI 以 `assets/report-template.html` 为基础。仅替换数据、文案、图片、链接和主题变量，不改变信息顺序。

HTML 文件是默认最终交付，聊天摘要不能替代它。必须完成“写入文件 -> 打开/预览 -> 桌面与手机检查 -> 回传可点击路径”后才能结束任务。

## 必备章节

1. **首屏结论**
   - 真实头像、账号名、类别/核心内容。
   - 一句直接结论，说清核心增长策略，不写概念性口号。
   - 3-4 个公开数据。
   - 三张结论卡：账号整体策略、最大优势、主要问题。
   - 一张代表封面，附“确定结论”。
2. **带日期的账号演化**
   - 按真实变化点分 2-4 个阶段，不机械平分。
   - 每阶段显示：时间、代表内容/数据、当时做法、变化原因、保留/放弃。
   - 末尾用一句结论说明最关键转折。
3. **内容诊断**
   - 固定 5 项：定位与人群、选题与需求、标题与封面、内容价值、主页承接。
   - 每项固定字段：现状、证据、判断、动作。
   - 使用“强 / 中 / 弱”仅作扫读标记，不伪造精确分数。
4. **爆款资产复用（含真实封面审计与可复制公式）**
   - 按“选题、标题、封面”三个维度指出首个高热样本和后续重复的热点选题、标题句式、拍摄场景、封面结构或用户需求。
   - 先写结论：爆款可重复，重复后数据会波动。
   - 使用简短图表 + 5-10 条样本表；不只列数据。
   - 在本章节内展示 3-5 张真实封面，包含高热、中段和低效样本；每张给定性标签和可见互动。
   - 在本章节内至少做一组高热 vs 低效视觉对比，说明具体封面差异。
   - 在本章节内输出可复制公式，说清爆款选题公式、爆款标题公式和爆款封面公式。
   - 在本章节内区分“应复制 / 应增加 / 不照搬 / 如何评估”。
5. **合规边界**
   - 只列与该账号相关的 3-4 项，不复制整本规范。
6. **对标学习与发展路线**
   - 先写迁移公式：`固定人群与需求 × 可持续拍摄场景 × 可重复内容结构 × 每篇新信息`。
   - 双栏展示：向博主学什么 / 初期先打造什么。
   - 三阶段展示：起号基建、验证与固化、放大与扩展。每阶段必须有目标、必做、保持不变/进阶信号。
   - 最后给内容配比和 3-5 个标题样例，标题不得代替发展路线。
7. **数据说明与来源**
   - 标注采集日期、公开/用户提供、未获取字段、日期推定方式和官方规范来源。

## 外链契约

报告必须能从“结论”直接回到“原内容”。以下对象出现时必须带可点击原始链接：

1. 首屏账号名、头像或“打开账号”：链接到该博主的小红书主页。
2. 账号演化：每阶段至少 1 条代表笔记链接，使用“查看代表笔记”。
3. 爆款资产复用：首个高热样本、每条后续复用样本、真实封面卡片和高低效对比样本均链接到对应笔记详情页。
4. 发展路线与真实工作流：凡写“参考某博主/某条内容/照此拆解”的地方，附“查看案例”链接；纯方法步骤无需虚构链接。
5. 数据来源：官方规则、第三方资料和用户提供的公开网页均使用来源名称作为链接文本。

生成规则：

- 采集阶段保存 `account_url` 和每条样本的 `note_url`，生成 HTML 时直接映射，不凭标题猜 URL。
- 外链统一使用 `target="_blank" rel="noopener noreferrer"`，链接文案使用“打开账号 / 查看原笔记 / 查看案例 / 查看来源”，不显示冗长裸 URL。
- 链接未获取时显示灰色文字“链接未获取”，不得渲染空按钮、`href="#"`、搜索页替代链接或错误主页链接。
- 同一笔记在同一卡片中只保留一个主链接，避免封面、标题、按钮三次重复造成视觉噪音；封面卡允许整卡可点击。

## 结论表达

- 每章顶部先给粗体结论，再放证据。
- 核心结论使用“确定结论”或“合理推断”标明证据等级。
- 不使用“母版”，改用“爆款拍摄场景 / 封面资产 / 内容结构”。
- 不写空泛的“提高质量”，动作要指明改什么、保留什么。
- 同一数据不在多章重复。证据不足的模块缩短或省略，不用空卡填满。

## 主题色系

1. 实际查看 3-5 张代表封面和头像。
2. 选择一个能代表博主的主色/强调色，不追随单张照片的偶然色。
3. 主色只用于链接、重点数据、导航状态和结论提示；大面积背景使用中性白/灰。
4. 主色过浅时，生成同色系的深色变体作文字强调，保证可读性。
5. 无法看见封面或无明确主色时，默认使用小红书红 `#e52b47`。
6. 不使用渐变、发光色块或与博主风格无关的装饰色。

## UI 结构

- 字体：系统无衬线字体，不调用外部字体。
- 容器：桌面侧边栏 220-240px，主内容最大 1180px；内容左对齐。
- 首屏：左文右图，标题最多 2 行，确定结论覆盖在图片下方边缘。
- 数据：使用无浮层的横向数据带，不把每个数据做成彩色卡片。
- 时间线：桌面 3 列、手机 1 列；日期使用强调色。
- 诊断：桌面 2 列，第 5 项通栏；手机全部 1 列。
- 封面：使用 3:4 固定容器 + `object-fit: contain`，永不裁切；图片下方放数据和结论。
- 发展路线：先双栏“学什么 / 初期打造什么”，再 3 列阶段卡，最后才放标题样例。
- 卡片：圆角不超过 8px，不嵌套卡片；只用在诊断、封面样本和阶段信息。
- 响应式：`<=980px` 隐藏侧边栏并显示顶部导航；`<=620px` 所有多列降为 1 列。
- 交互：仅保留锚点导航、链接 hover/按下状态和当前章节高亮；不添加无信息价值的动画。

## 可移植性

- HTML/CSS/JS 必须自包含，不依赖 npm、CDN、网络字体或创建者电脑路径。
- 图片优先下载到报告同级 `assets/<account-slug>/`，HTML 使用相对路径。无法下载时使用原链接和文字占位，不崩溃。
- 账号和笔记保留可点击小红书链接。
- 输出文件与资源路径只使用当前工作区相对路径。
- 无本地服务时，报告必须可通过 `file://` 直接打开。

## 交付前验证

- 验证桌面端 1280px 和手机端 390px：无横向溢出、重叠、文字裁切。
- 检查所有图片成功加载，真实封面保持 3:4 完整显示。
- 检查首屏在桌面端能看到结论、主要数据和代表封面。
- 检查 7 个必备章节均存在；证据不足的章节可缩短，但不得伪造内容。
- 逐项检查账号主页、演化代表笔记、爆款复用表（含真实封面和高低效对比）、发展路线案例和来源链接；外链必须是有效 `http(s)` 地址并在新标签页打开。
- 搜索 HTML 中的空 `href`、`href="#"`、未替换的 `{{...URL...}}`；存在任一项即不允许交付。
- 能运行本地脚本时，执行 `python3 scripts/validate_report_links.py <report.html>`；校验失败必须修正后重跑。
- 如当前 App 不能预览/截图，执行“跨宿主能力协商”，询问是否有浏览器、HTML 预览、本地服务或文件打开能力。
